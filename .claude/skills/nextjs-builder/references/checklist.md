# 5軸レビューチェックリスト

実装の**前**（設計時）と**後**（提出前）に通す。1つでも欠けたら直すか、理由を明示する。

## 実装前（設計チェック）
- [ ] 既存規約を調べたか？（PM・Next バージョン・DB・認証・スタイリング・ESLint 設定）
- [ ] server / client の境界をどこに置くか決めたか？`'use client'` は葉に押し下げたか？
- [ ] データ取得の場所とキャッシュ/再検証戦略を決めたか？
- [ ] 信頼境界（Server Action / Route Handler）はどこか、そこで何を検証・認可するか決めたか？
      テナント/認可境界の分離方式（所有/ロール/`tenant_id`+RLS 等）を決めたか？
- [ ] 濫用されやすいエンドポイント（認証/メール/決済/公開API）にレート制限・クォータを設計したか？
- [ ] 依存の導入を分岐で判断したか？ 既存に選択があれば尊重、新規/能力未導入なら標準機能で足りるか確認のうえ
      5軸で選定し**理由を添え**、重大・不可逆な選定（DB/ORM/認証/デプロイ）と新規依存は**着手前に確認**したか
      （既存で足りるものを不要に依存化していないか）。

## 実装後（5軸レビュー）

### ① 可読性
- [ ] 命名が意図を語るか？マジック値は定数化したか？
- [ ] 早期リターンでネストが浅いか？関数/コンポーネントは短く単一目的か？
- [ ] コメントは「なぜ」に限られているか？死んだコードは無いか？
- [ ] 既存の Prettier/ESLint スタイルに合っているか？
- [ ] アクセシビリティ（セマンティック HTML・`label` 関連付け・キーボード操作・必要な ARIA）を満たすか？

### ② ドメイン適合（RSC イディオム）
- [ ] Server Component をデフォルトにし、client 化は必要な葉だけか？
- [ ] ルーティング予約ファイル（`page`/`layout`/`loading`/`error`/`route`）を正しく使ったか？
- [ ] メタデータは `metadata`/`generateMetadata` で扱ったか？
- [ ] ページ描画のために自分の Route Handler を fetch していないか？

### ③ パフォーマンス
- [ ] 独立データ取得を `Promise.all` で並列化したか？重複は `React.cache` か？
- [ ] 遅い領域を `<Suspense>` でストリーミングしたか？
- [ ] `next/image` / `next/font` を使ったか？重い client 依存を `next/dynamic` 化したか？
- [ ] クライアントに不要な大きな props / JS を送っていないか？
- [ ] ライブラリは標準機能で代替できないか確認し、名前付き import・サーバー隔離でバンドルを抑えたか（`references/libraries.md`）？

### ④ セキュリティ
- [ ] 全境界入力を Zod 検証したか？
- [ ] 濫用されやすいエンドポイント（認証/メール/決済/公開API）にレート制限を掛けたか？認証情報を自前で扱うなら argon2id/bcrypt ハッシュ・列挙対策をしたか？
- [ ] Server Action / Route Handler の先頭で認証・認可（所有チェック含む）したか？
- [ ] 非公開資源を表示する**読み取り経路**（Server Component / `lib/data`）でも認証＋所有/可視性チェックをしたか（IDOR 回避）？
- [ ] 秘密がクライアントに漏れていないか？秘密モジュールに `server-only` を付けたか？
- [ ] Client Component / RSC に渡す props を必要フィールドのみに絞ったか（DB オブジェクト丸ごとを渡していないか）？
- [ ] `dangerouslySetInnerHTML` / 文字列連結SQL / オープンリダイレクト / SSRF を避けたか？

### ⑤ SOLID
- [ ] 各コンポーネントは単一責任か？（取得・表示・状態を混ぜていないか）
- [ ] props は最小限か（god props でないか）？
- [ ] UI がデータ層の抽象越しに依存しているか（具体実装を直接埋め込んでいないか）？
- [ ] `any` を排し、不正状態を型で表現不能にしたか？

## 種別別チェック（該当する種別のみ）

### 要件/設計からの実装（スタック選定）
- [ ] 長文の仕様書を **ID 付き受け入れ基準**に分解し、完了時に**全 ID を突き合わせて網羅確認**したか（未対応は対応/保留＋理由を明示。矛盾・欠落は訊いたか）？
- [ ] 設計を左右する非機能要件（規模/性能・データ特性・リアルタイム/非同期・デプロイ先/ランタイム・認証要否）を把握したか（曖昧なら訊いたか）？
- [ ] 各カテゴリの候補を 5軸＋適合基準で評価し、標準機能で足りないことを確認したか？
- [ ] 各選定に理由と代替案を添えたか（「流行っているから」で選んでいないか）？
- [ ] スタック決定記録を着手前に 3〜6行で提示し、重大・不可逆な選定（DB/ORM/認証/デプロイ）と新規依存を着手前に確認したか？
- [ ] 不可逆な選定の理由を生成物側（README/ADR）へ短く記録したか？

### 機能追加
- [ ] 呼び出し元・型・データ層・近接ルートを読み、既存パターンを踏襲したか？
- [ ] 変更の影響範囲（波及先）を把握したか？
- [ ] 既存の公開インターフェース/型を壊していないか？

### リファクタリング
- [ ] **外から見た振る舞いを変えていない**か？（入出力・副作用が同一）
- [ ] 依頼スコープを勝手に広げていないか？（「ついで」の改変を混ぜていない）
- [ ] 型/リンタ/ビルド（あればテスト）で保全を確認したか？
- [ ] 1コミット1目的か？（整形・改名・ロジック変更を分けたか）

### バグ修正
- [ ] バグを**再現**できたか？
- [ ] 症状ではなく**根本原因**を特定したか？
- [ ] 最小スコープの修正か？（無関係な変更を混ぜていない）
- [ ] 再現手順で**直ったことを検証**したか？
- [ ] 可能なら回帰テストで固定し、同種のバグが近傍に無いか確認したか？（テスト手法は `references/testing.md`）
